- (event) Create intent confirmed; initialized product brief workspace with placeholder topic until discovery identifies the product.
- (decision) Product identified as Hexalith Agents: manages AI agents that interact with external actors from Hexalith.Parties through Hexalith.Conversations; activation can be conversation-change-triggered or explicit mention only.
- (decision) Agent management scope includes identity, instructions, model choice, tools, memory, permissions, lifecycle, audit, and activation rules; agents can act automatically or only after human confirmation.
- (decision) V1 target: agent administrators configure agents; agents only reply in conversations; primary use case is replying when explicitly called in a conversation; human validation can be assigned to the conversation owner, the party that explicitly called the agent, or a predefined party list.
- (decision) Brief purpose is broad: internal alignment, PRD input, architecture input, investor-style pitch, and implementation planning; frame as an executive product brief that feeds downstream artifacts rather than replacing them.
- (event) Local source scan found no prior Agents planning artifact; root README is a stub. Conversations README establishes AIAgent participants, PartyId attribution, fail-closed participant validation, tenant-scoped conversations, and content-safe typed errors that should constrain Agents.
- (event) External scan: OpenAI Agents SDK frames agents as app-owned orchestration/tool execution/approvals/state; OpenAI and Microsoft document HITL approval/resume flows; Anthropic tool use emphasizes app-executed client tools and model-controlled trigger boundaries. Relevance: Hexalith Agents should emphasize governed conversation participation, approval policy, and observability rather than merely LLM calls.
- (decision) User selected coaching path: shape the product brief section by section through discussion and pressure-test assumptions before drafting.
- (decision) Problem statement: Hexalith already has a multi-party conversation module, but lacks governed AI agents that can participate and respond in those conversations.
- (decision) Primary consequence for V1: conversation participants cannot explicitly call a named AI assistant inside a conversation when they need help.
- (decision) First named agent is hexa: a general-purpose agent intended to be the main AI agent called by Parties inside conversations.
- (decision) hexa V1 primarily answers questions using conversation context. V2 expands to configured tools and memories for domain-question answering.
- (decision) Trust model: in auto mode hexa posts directly to the conversation; in confirmation mode hexa creates a proposed reply that authorized parties approve before posting.
- (decision) V1 confirmation workflow supports editing a proposed reply before approval and requesting regeneration before approval; proposal lifecycle must capture more than approved/rejected.
- (decision) Identity decision: every agent has a PartyId, so AI agent participation in conversations is attributable through Hexalith.Parties identity rather than an Agents-only identifier.
- (decision) V1 agent creation: agent administrators create an agent and the Agents module creates or links the corresponding PartyId through Hexalith.Parties as part of the workflow.
- (decision) Scope boundary: memories and tools are out of V1. V1 activation is explicit call in Conversations only. V2 uses Hexalith.Memories for memory management and adds activation on Project content changes, Folder content changes, and Conversation changes via Hexalith.Projects, Hexalith.Folders, and Hexalith.Conversations.
- (decision) V1 success criteria: create hexa with Party identity and response policy; Party can explicitly call hexa in a conversation; hexa answers from conversation context as attributed AI participant; confirmation flow supports correct approver edit/regenerate/approve; unauthorized parties cannot call/approve/configure; every reply has audit evidence for caller, agent, source conversation, approval path, and final response.
- (decision) Context/audit boundary: in V1 the agent can see the full conversation. In V2, context may use a configurable compacted conversation summary. Non-approved answers are not part of the Conversation; a separate approval process manages approval, editing, regeneration, and audit evidence. Approved answers are then added to the conversation. Tenant isolation must be strict.
- (event) Drafted first complete product brief from memlog decisions, focused on governed explicit invocation of hexa in Conversations with V1/V2 boundaries and open questions.
- (event) Finalized product brief: status set to final; structure and prose standards applied directly; no addendum needed; no external handoffs configured.
