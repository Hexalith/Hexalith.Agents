# Implementation Readiness Review — agents

## Overall verdict

The UX spine pair is not implementation-ready as a complete source for architecture and story-dev. It gives strong FrontComposer inheritance, visual semantics, accessibility, and proposal-state language, but core V1 workflow decisions remain open in places where downstream teams would otherwise have to reread product sources or guess. Configuration, status, and audit surfaces are close; Conversation invocation and proposal workflow stories should be blocked until the contracts below are closed.

## Findings

- **[critical]** Conversation-originated invocation is explicitly unresolved even though it is the primary runtime entry point and drives UX flow, API route naming, source ownership, and acceptance tests (`_bmad-output/planning-artifacts/ux-designs/ux-agents-2026-06-23/EXPERIENCE.md:52`). *Fix:* Choose the V1 invocation pattern, or a constrained set of patterns, and add the owning surface, required fields, permission checks, failure states, API naming impact, and a low-fi Conversation wireframe before creating Agent Call stories.
- **[critical]** Proposed Agent Reply runtime and storage ownership is still an open question, so architecture cannot assign aggregate boundaries, API contracts, status ownership, or proposal queue/editor responsibility without a product/architecture decision (`_bmad-output/planning-artifacts/ux-designs/ux-agents-2026-06-23/EXPERIENCE.md:290`). *Fix:* Add an ownership contract that names the module/aggregate responsible for proposal state, versions, expiry, authorization decisions, audit evidence, and boundaries with Conversations, Parties, Providers, admin UI, and public API.
- **[high]** Open questions are copied into the UX spine without blocker vs carry-forward triage, while the PRD gives several revisit gates before UX flow specification, aggregate design, provider contracts, proposal lifecycle stories, audit stories, and context-building stories (`_bmad-output/planning-artifacts/ux-designs/ux-agents-2026-06-23/EXPERIENCE.md:283`). *Fix:* Replace the flat open-question list with a triage table: blocker/carry-forward, owner, downstream gate, affected surfaces, and minimum assumption if deferred.
- **[high]** Admin/API parity is stated as a goal but not made into a build contract, despite the PRD requiring the same authorization outcomes through admin UI and API/client contracts (`_bmad-output/planning-artifacts/prds/prd-agents-2026-06-23/prd.md:320`). *Fix:* Add a parity matrix for Provider administration, Agent configuration, Agent invocation, proposal workflow, status, and audit that maps UI action, API/client operation, shared authorization result, status/error shape, and audit evidence.
- **[high]** Component contracts stop at behavioral descriptions; story-dev still lacks per-component inputs, actions, events, ownership, auth/read-only variants, and required data fields for the grids, badges, editor, policy builder, and audit panel (`_bmad-output/planning-artifacts/ux-designs/ux-agents-2026-06-23/EXPERIENCE.md:82`). *Fix:* Add a component contract table for each named component with owner, consumed data, emitted commands, loading/empty/error/denied states, accessible name requirements, and linked API/status dependencies.
- **[high]** Expiry and approver notification are used by the proposal flow but remain unresolved at the exact point where proposal lifecycle and notification stories would need acceptance criteria (`_bmad-output/planning-artifacts/prds/prd-agents-2026-06-23/prd.md:496`). *Fix:* Define the V1 expiry default/configurability, countdown/warning display, terminal transition behavior, and notification path, or explicitly mark queue-only discovery as the launch assumption.
- **[medium]** FrontComposer inheritance is directionally correct but misses implementation-critical constraints from the inherited project context, including the exact Fluent package pin, raw HTML control prohibition, custom icon factory, and multi-section FluentAccordion guideline (`Hexalith.FrontComposer/_bmad-output/project-context.md:29`). *Fix:* Add a short FrontComposer implementation appendix so stories can use the UX spine without rereading source or project context for these non-obvious rules.
- **[medium]** Mock/wireframe coverage is still undecided, and the most complex surfaces are not sketched even at low fidelity (`_bmad-output/planning-artifacts/ux-designs/ux-agents-2026-06-23/.memlog.md:20`). *Fix:* Add minimal wireframes or explicit no-mock rationale for Conversation invocation, proposal detail/editor with version history, approver policy builder, provider catalog, proposal queue, and narrow-width review-only behavior.

## Positive coverage

- FrontComposer and Fluent visual inheritance is clear, and the design correctly avoids creating a separate Agents design system (`_bmad-output/planning-artifacts/ux-designs/ux-agents-2026-06-23/DESIGN.md:115`).
- Status color semantics are precise and preserve important distinctions such as generated vs posted, approved vs posting pending, and failure vs blocked dependency (`_bmad-output/planning-artifacts/ux-designs/ux-agents-2026-06-23/DESIGN.md:121`).
- Shell/domain ownership is mostly clear: FrontComposer owns chrome and shell behavior; Agents owns registered nav entries, page bodies, domain copy, and BFF/API-facing behavior (`_bmad-output/planning-artifacts/ux-designs/ux-agents-2026-06-23/EXPERIENCE.md:27`).
- The proposal lifecycle vocabulary is strong and protects the key product invariant that unapproved generated content is not a Conversation Message (`_bmad-output/planning-artifacts/ux-designs/ux-agents-2026-06-23/EXPERIENCE.md:136`).
- Accessibility and responsive fail-closed rules are explicit enough to guide initial acceptance criteria (`_bmad-output/planning-artifacts/ux-designs/ux-agents-2026-06-23/EXPERIENCE.md:176`).

## Residual risks

- If OQ-1 resolves to multiple invocation affordances, the IA and component contracts may need another pass before story slicing.
- If architecture moves provider governance or proposal storage outside Hexalith Agents, the UX ownership and parity matrices must be updated before implementation.
- State token names can be refined by architecture, but the UX distinctions in readiness, proposal, audit, posting, and failure states must remain invariant.
