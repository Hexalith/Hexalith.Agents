# Accessibility Review — agents
## Overall verdict
The UX spine has a solid accessibility floor: it inherits FrontComposer FC-A11Y, requires visible text plus accessible names for status, preserves keyboard access for proposal actions, and explicitly blocks secrets/raw payloads from assistive surfaces. It is not yet fully sufficient for downstream architecture/story-dev because several high-risk workflows rely on broad principles instead of deterministic focus, selection, live-region, and responsive-denial contracts.
## Findings
- **[high]** Conversation invocation remains unresolved, so keyboard entry, focus placement, accessible naming, and announcement behavior for calling `hexa` cannot be implemented consistently across mention/command/action candidates (EXPERIENCE.md:52). *Fix:* Close OQ-1 with accessibility acceptance criteria for each allowed invocation pattern: keyboard path, focus target after invoke/cancel/error, localized accessible name, response-mode announcement, and sanitized denial/status text.
- **[high]** Proposal editor says it must be keyboard-operable but does not define the accessible selection model for generated/edited/regenerated versions, even though approval posts exactly the selected version (EXPERIENCE.md:183). *Fix:* Specify a deterministic focus order and semantics: version list as tabs/listbox/grid with selected/current state, editor labelled by selected version, compare metadata reachable before approve, and approval confirmation naming the exact version timestamp/source.
- **[high]** Focus return assumes the triggering row/action still exists, but approval, rejection, abandonment, expiry, filtering, or "needs my action" removal can delete the trigger from the current view (EXPERIENCE.md:171). *Fix:* Define fallback focus targets: next eligible row, queue status summary, filter-empty reset, or proposal-detail heading, with a polite announcement when the triggering item leaves the view.
- **[medium]** Grid semantics are named, but responsive row-detail/column-prioritization behavior does not say how hidden columns preserve header relationships, row action names, and keyboard expand/collapse semantics (EXPERIENCE.md:182). *Fix:* Add FC-TBL acceptance criteria for collapsed columns: keyboard-expandable row details, retained header labels for hidden data, safe row action labels, exposed sort/filter state, and no loss of "needs my action" or expiry context.
- **[medium]** Live-region guidance lists transitions but does not map each transition to politeness, atomicity, de-duplication, or source-specific announcement text (EXPERIENCE.md:184). *Fix:* Define a transition matrix: polite for created/posted/failed/expired/denied status changes, assertive only for immediate destructive action prevention, atomic summaries for multi-field status changes, and no repeated announcements for ordinary pending progress.
- **[medium]** Responsive fail-closed approval requires a visible reason, but not an accessible disabled-action reason for keyboard and screen-reader users when approval/editing controls are unavailable on narrow viewports (EXPERIENCE.md:211). *Fix:* Require unavailable high-impact actions to expose a localized reason through focusable explanatory text or `aria-describedby`, not only a disabled control or visual message.
- **[medium]** Localization covers domain copy, but accessible names, tooltips, aria descriptions, and live-region announcements are not explicitly included in the whole-string localization contract (DESIGN.md:139). *Fix:* Extend FC-L10N requirements so all non-visible assistive text uses localized whole strings with named placeholders and redacted/safe data only.
## Positive coverage
- FrontComposer ownership is clear: shell skip links, keyboard shell behavior, landmarks, live regions, and localization split are inherited rather than reinvented (EXPERIENCE.md:27; .memlog.md:8).
- Status treatment is accessibility-aware: color-only status is prohibited, badges require visible text and accessible names, and status grids must expose table semantics (DESIGN.md:131; EXPERIENCE.md:181).
- Approval safety is strongly modeled: generated proposals are not Conversation Messages, approval applies to a selected version, terminal states cannot post, and responsive approval fails closed when context is insufficient (EXPERIENCE.md:136; EXPERIENCE.md:170; EXPERIENCE.md:211).
- Secret and raw payload protection is explicit across UI, audit, accessible names, tooltips, diagnostics, and announcements (DESIGN.md:212; EXPERIENCE.md:187; prd.md:432).
## Residual risks
- OQ-3 proposal expiry and OQ-4 approver notification will need accessibility-specific acceptance criteria for announcement timing, stale proposal focus behavior, and notification dismissal.
- Actual compliance still depends on FrontComposer/Fluent component implementation choices, especially custom proposal/editor overrides that bypass generated FC-A11Y behavior.
- Generated conversation/proposal content may be safe to display but unsafe as an accessible name, tooltip, diagnostic, or announcement; story acceptance should test that separation directly.
