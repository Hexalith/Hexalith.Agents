# Governance Review — agents
## Overall verdict
The UX spine captures the right governance intent, especially around proposed-reply separation, provider-secret safety, version preservation, and safe status copy. It is not yet sufficient as a downstream implementation contract for every security-sensitive path because several high-impact authorization and evidence boundaries are still open or underspecified. Treat the current spine as a strong draft that needs blocking governance amendments before architecture and story-dev rely on it.

## Findings
- **[critical]** Approver authority can become broader than Source Conversation access; the policy builder lists tenant roles and predefined Parties but does not require proposal discovery/detail/edit/approval to also pass current Source Conversation access, so generated content could leak to a same-tenant Party that cannot inspect the Conversation (_bmad-output/planning-artifacts/ux-designs/ux-agents-2026-06-23/EXPERIENCE.md:89). *Fix:* Add a contract that proposal list/detail/edit/regenerate/approve require both Approver Policy authorization and fresh Source Conversation access, with blocked/hidden/denied states for missing or stale access.
- **[critical]** Conversation-originated invocation remains explicitly unresolved, which leaves identity attribution, mention membership, prompt capture, denial copy, and proposed-vs-posted rendering open to implementation interpretation (_bmad-output/planning-artifacts/ux-designs/ux-agents-2026-06-23/EXPERIENCE.md:52). *Fix:* Make OQ-1 a blocking UX governance gate before invocation stories: choose the V1 pattern and specify visible `hexa` identity, Source Conversation binding, call authorization, in-flight state, and failure/no-post states.
- **[high]** The state model authorizes a call once but does not define action-time reauthorization when Conversation access, Party identity, Agent lifecycle, provider state, or approver policy changes while a proposal/editor is open (_bmad-output/planning-artifacts/ux-designs/ux-agents-2026-06-23/EXPERIENCE.md:123). *Fix:* Add `revalidating`, `authorization stale`, and `denied after revalidation` states and require re-checks immediately before generation, regeneration, approval, posting, and automatic posting.
- **[high]** Proposal discovery and notification are not privacy-bounded; the queue supports pending and historical proposal discovery, while the notification path is unresolved, but neither surface defines safe payload rules for generated text, conversation snippets, caller names, or denial-safe counts (_bmad-output/planning-artifacts/ux-designs/ux-agents-2026-06-23/EXPERIENCE.md:91). *Fix:* Define queue-row and notification payload tiers: opaque reference only until authorization is proven, no generated content/snippets in notifications by default, and filtered-empty/permission-denied states that do not reveal unauthorized records.
- **[high]** Audit availability is modeled as a display state, not as a side-effect success invariant, so downstream stories could report posted/approved success while evidence is missing or uncorrelated (_bmad-output/planning-artifacts/ux-designs/ux-agents-2026-06-23/EXPERIENCE.md:156). *Fix:* Require every side-effecting success state to show a durable audit/correlation reference, or define an explicit `posted with audit pending` incident state with retry/escalation and no silent success.
- **[high]** API/admin parity is a journey and success criterion, but the UX contract lacks an operation-by-operation parity matrix for states, denials, errors, and audit references (_bmad-output/planning-artifacts/ux-designs/ux-agents-2026-06-23/EXPERIENCE.md:271). *Fix:* Add a parity table mapping each admin action to the public API/client operation, shared authorization outcome, structured error shape, and audit/status evidence expected.
- **[medium]** Version history preserves generated, edited, and regenerated versions, but it does not require immutable version identifiers, content fingerprints, or stale-selection protection before approval (_bmad-output/planning-artifacts/ux-designs/ux-agents-2026-06-23/DESIGN.md:198). *Fix:* Require every version row and approval confirmation to expose a stable version id/reference, source, timestamp, and selected-version lock; block approval if a newer version appears after selection.
- **[medium]** Party identity attribution is conceptually covered, but the UX does not require the posting/approval surfaces to show a stable Agent Party reference or handle display hydration failure without confusing `hexa` with the caller or a generic system account (_bmad-output/planning-artifacts/ux-designs/ux-agents-2026-06-23/EXPERIENCE.md:109). *Fix:* Add display rules for `hexa` name plus stable PartyId/support-safe reference in confirmations and audit, and fail closed for posting when command-time Party validation fails.
- **[medium]** Provider secret safety says secrets are not displayed, but the provider administration UX does not define rotation, replacement, clearing, validation failure, copy/export, or accessibility behavior for write-only secret fields (_bmad-output/planning-artifacts/ux-designs/ux-agents-2026-06-23/DESIGN.md:188). *Fix:* Add secret field states for `not configured`, `configured`, `replace pending`, `rotation failed`, and `cleared`, with redacted validation errors and no secret value in DOM text, tooltips, copied text, logs, or audit.
- **[medium]** Automatic Response Mode is presented as a normal mutually exclusive mode, but the UX does not require an activation confirmation that this mode posts generated content without human approval (_bmad-output/planning-artifacts/ux-designs/ux-agents-2026-06-23/EXPERIENCE.md:87). *Fix:* Add an activation/reconfiguration review step that states the tenant, Agent Party, provider/model, call eligibility, and "posts without approval" consequence, with future-only effect and audit reference after save.

## Positive coverage
- Proposed Agent Replies are repeatedly distinguished from Conversation Messages, including pending, approved, posting pending, posted, and posting failed states.
- Provider secrets, raw payloads, stack traces, and unrelated tenant data are explicitly excluded from UI, audit, accessible names, tooltips, diagnostics, and announcements.
- Readiness and status semantics cover missing Party identity, provider unavailability, context blocking, authorization denial, failed generation, failed posting, and terminal proposal states.
- Version preservation is a core contract: generated, edited, and regenerated versions remain visible and approval applies to a selected version.
- FrontComposer inheritance brings policy-gated navigation, table states, localization ownership, focus behavior, live regions, and fail-closed responsive rules into the UX spine.

## Residual risks
- Proposal expiry, approver notifications, latency targets, cost controls, provider capability metadata, audit retention/export/deletion/legal hold, safety filters, and oversized Conversation Context remain open questions.
- Global Providers ownership and platform-scoped versus tenant-scoped provider configuration still need architecture closure before provider administration stories are safe.
- Generated FrontComposer UI will need governance-specific overrides and tests for authorization boundaries, stale/degraded states, redaction, and admin/API parity.
