## 2. What Went Well

**The Epic 2 step-pattern scaled.** Epic 3 reused the orchestrator -> single trusted command -> pure aggregate handler -> twin policy pattern across create, edit, regenerate, approve/post, and terminal-state transitions. This kept side effects out of aggregates and made no-drift tests practical.

**The proposal lifecycle stayed append-only.** Generated, edited, and regenerated versions were preserved as immutable versions. Approval freezes one selected `VersionId`; terminal states preserve history and prevent later posting. Story 3.7 then rendered the full version history without treating proposals as Conversation Messages.

**Fail-closed behavior was consistent.** Deferred gateways and ports return `NotAuthorized`, `Unavailable`, or blocked states instead of partial success. Proposal queue/detail pages do not render ready bodies on denied, unavailable, stale, or not-found reads. Cross-tenant and unauthorized disclosure is avoided through empty lists, zero counts, safe error states, and content-free accessible names.

**Accessibility and UI contract discipline matured.** Story 3.7 closed the hosted-control responsibilities deferred by 3.3-3.6: focusable route heading, keyboard-selectable version history, compare-before-approve, Esc without commit, selected-version lock, terminal "start a new call" routing, and a live-region politeness matrix.

**QA automation found real coverage gaps and closed them.** Story 3.1 added full command-chain proposal lifecycle tests. Story 3.4 added failure/no-drift edge tests. Story 3.5 added approval lifecycle gap tests. Story 3.6 added terminal cross-action guards and UI presentation coverage. Story 3.7 added live-region, focus, compare, expiry, and version-selection tests.

## 3. What Was Hard

**The live read/runtime path stayed deferred by design.** Story 3.2 and 3.7 shipped stable read contracts and UI gateways, but no server projection, query handler, BFF/client binding, live `NeedsCurrentUserAction`, or audit-producing read authorization. This was intentional and fail-closed, but it meant Epic 3 produced a strong proposal surface over placeholder data seams.

**The retryability model was easy to get subtly wrong.** The review of Story 3.4 fixed edit/regenerate asymmetry so regenerated proposals remain editable. Story 3.7 still carried a known trap: after `ProposalEditFailed` or `ProposalRegenerationFailed`, coarse status can block editing even when proposal sub-state is retryable. The right design rule is to gate affordances on proposal sub-state (`Pending`, `Edited`, `Regenerated`), not coarse interaction status.

**Toolchain friction repeated.** Restore sometimes failed with no surfaced MSBuild errors. VSTest hit `SocketException (13)`, so story records repeatedly switched to built xUnit v3 executables. Serialized Release builds with `-m:1` remained reliable, but the workflow cost kept recurring.

**Record accuracy remained the process weak spot.** Epic 2 already committed to fixing stale Dev Agent Record counts and File List omissions. Epic 3 still had review corrections in 3.2 and 3.7, and similar drift appears in other story notes. The checklist did not eliminate the issue.

## 7. Significant Discoveries

**No Epic 4 scope revision was required.** Epic 3 validated the planned proposal workflow and gave Epic 4 the contracts/UI surfaces it needed.

**A process discovery is significant:** Dev Agent Record accuracy cannot be fixed by adding a reminder to story tasks. It needs a mechanical pre-review gate that compares recorded counts and File List entries against actual test output and `git status --short`.

**A domain discovery should be recorded:** proposal action eligibility must be defined primarily by proposal sub-state, not by coarse `AgentInteractionStatus`. This avoids retryability traps after edit/regeneration failures.

**A production-readiness discovery is now explicit:** V1 is proposal-workflow complete and later contract-complete, but not live. The remaining work is binding deferred seams under governance decisions and adding live integration tests when each seam is bound.
