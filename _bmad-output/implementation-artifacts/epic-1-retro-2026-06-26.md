## 2. Deep Story Analysis - Synthesized Patterns

### What Went Well

**The team built readiness incrementally without reshaping earlier work.** Each story appended one gate to the same `Agent` aggregate and status contract: base lifecycle, Party identity, provider/model, response mode, approver policy, then content safety. Existing enum ordinals were preserved, new fields were additive, and earlier activation tests were updated when a new gate correctly made an older "ready" fixture incomplete.

**Fail-closed verdicts became a repeatable safety pattern.** External readiness checks were fed into pure aggregate handlers through server-populated extensions: `party:linkValidation`, `provider:selectionValidation`, and `approver:policyValidation`. Reviews hardened those parsers to exact canonical enum names so numeric or aliased values fail closed to `Unknown`.

**Deferred seams were explicit rather than accidental.** Live command dispatch, ProviderCatalog reading, approver resolution, UI read gateways, and AppHost topology were deferred behind named placeholders. The placeholders return safe unavailable/not-authorized behavior or throw clear "not wired yet" errors. That let Epic 1 build a strong foundation without pretending integration was already live.

**Sensitive data boundaries held.** Agent instructions, content-safety policy text, provider configuration references, and Party PII stayed out of status views, rejections, accessible names, logs, and test IDs. The public contracts remained provider-SDK-free and secret-value-free.

**Story 1.8 turned the setup state into a usable operator surface.** The UI made the important distinction explicit: active lifecycle is not the same as callable readiness. Success is only used when the Agent is active and has no blockers.

### What Was Hard

**Story 1.8 was broad, and the risk was real.** It bundled navigation, overview, provider grid, configuration form, approver policy, badges, localization, and accessibility. The team compensated with one bUnit suite per acceptance criterion and a senior review that found a meaningful AD-12 issue.

**Unknown dependency state was easy to overstate.** The review fix in Story 1.8 changed approver-policy source rows from affirmative "Available" to neutral "not yet evaluated." That is the clearest Epic 1 lesson: when the live read path is deferred, the UI must not render uncertainty as readiness.

**Every new gate forced fixture churn.** Activation tests from earlier stories had to be revisited as Party, provider, response-mode, approver, and content-safety gates landed. That churn was expected and healthy, but it is a signal that readiness fixture helpers must be treated as shared infrastructure, not copied per test.

**Dev Agent Record drift started here.** Several senior reviews corrected stale test counts or incomplete File Lists. Later retrospectives show this became a repeated process failure, so Epic 1 is where the pattern first became visible.

### Breakthroughs Worth Repeating

- The `ProviderCatalog` replay-derived `CapabilityVersion` solved AD-4 snapshot needs without reshaping Story 1.2 events.
- The content-safety gate was stronger than a verdict gate: because it is pure Agent state, direct activation cannot bypass it by omitting or forging an external verdict.
- The FrontComposer setup surfaces used the existing display contracts instead of inventing UI-only DTOs, preserving AD-15 UI/API parity.

## 7. Significant Discoveries And Plan-Update Assessment

No significant discovery invalidates Epic 2. The main discoveries are operational:

- **Readiness has two layers.** Durable state can say which setup fields exist, but live callability still depends on fresh external verdicts. The UI and status contracts must keep those layers separate.
- **Deferred seams are now part of the architecture, not temporary clutter.** They are safe only while they remain explicit, tested, and mapped to live owners.
- **Record accuracy needs enforcement.** Epic 1 already shows stale File List/test-count drift, and later retrospectives confirm the pattern continued.

**Epic update required:** No.

---

## 8. Key Takeaways

1. Epic 1 succeeded because setup readiness was built as additive gates on a pure `Agent` aggregate, not as scattered UI or orchestration state.
2. The server-populated verdict-extension pattern is useful, but every verdict parser must be exact-name and fail-closed.
3. Unknown dependency state must stay visibly unknown; Story 1.8 proved how easy it is to accidentally render deferred data as available.
4. Epic 2 can proceed, but live invocation/posting must be treated as a separate binding concern, not as an implied result of completed setup.
