# Story 2.3: Build Conversation Context With Safe Bounds

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a Conversation Participant,
I want `hexa` to use the authorized Conversation context safely,
so that the answer is grounded in the Conversation without silent truncation or unrelated data exposure.

## Acceptance Criteria

**AC1 - Load only authorized Conversation detail + visible timeline through the Conversations boundary; V1 excludes all non-conversation sources**
**Given** an AgentInteraction passes invocation gates
**When** context building starts
**Then** the system loads only authorized Conversations detail and visible timeline content through supported Conversations client/API boundaries
**And** V1 excludes long-term memory, project content, folder content, file content, tool output, external-channel content, and non-conversation retrieval.

**AC2 - Full-context fit records full-context usage + policy version + budget metadata + safe evidence, raw context redacted**
**Given** the full Source Conversation fits the selected Provider/model context budget after reserving configured output tokens
**When** context building completes
**Then** the interaction records full-context usage, context policy version, model budget metadata, and safe context evidence
**And** raw context is protected or redacted from logs, telemetry, queue summaries, and status badges.

**AC3 - Oversized or not-fresh-enough context with no approved bounded behavior fails closed with no side effects**
**Given** the Source Conversation exceeds the safe context budget or cannot be loaded fresh enough
**When** no approved bounded-context behavior is configured
**Then** the system records a context-blocked failure
**And** no provider call, Proposed Agent Reply, or Conversation Message is created.

**AC4 - Approved bounded-context behavior records behavior + policy version + bounds + audit-safe metadata; never silent truncation**
**Given** an approved bounded-context behavior exists
**When** it is used
**Then** the behavior, policy version, bounds, and audit-safe metadata are recorded
**And** the system never silently truncates context without explicit policy evidence.

### Why No `ContextLoading` Status

Story 2.1's `AgentInteractionStatus` doc comment forward-listed "context-loading" as a possible later state, and the UX state table shows a `context loading` Agent-Call state. **This story deliberately does NOT add a durable `ContextLoading` status.** The architecture sequence records only two outcomes back to the interaction ŌĆö `RecordContextReady` or `ContextBlocked` (ARCHITECTURE-SPINE.md lines 240-243) ŌĆö exactly like the Story 2.2 gate transitioned `Requested ŌåÆ Authorized | Denied | Blocked` with no transient "authorizing" status. "Context loading" is the **transient UI representation** of an `Authorized` interaction whose context build is in flight (the orchestrator is mid-read before it dispatches the single context command); it is surfaced by Story 2.6 from the in-flight call state, not persisted as an aggregate status. Adding a durable loading status would require an extra started-event with no architectural backing and would complicate idempotency. If a later story needs a durable in-flight marker, it can be appended additively then. [Source: ARCHITECTURE-SPINE.md sequence lines 237-243; AgentInteractionStatus.cs:5-8; UX EXPERIENCE.md state table]

### Context Budget Rules (AC2/AC3/AC4 ŌĆö the pure decision)

The orchestrator assembles one `AgentInteractionContextMeasurement`; the aggregate decides on it. **Reserved output tokens = the catalog entry's `MaxOutputTokenLimit`** (the configured output ceiling for the selected model ŌĆö AC2 "after reserving configured output tokens"). `availableContextBudget = ContextWindowTokenLimit ŌłÆ ReservedOutputTokenCount`.

| Condition (evaluated in order) | Outcome | Status | Evidence |
| --- | --- | --- | --- |
| `LoadOutcome` is `Unauthorized`/`Unavailable`/`Unknown` | block | `ContextBlocked` | reason `ContextUnavailable`; numerics `0` |
| `LoadOutcome` is `Stale` | block | `ContextBlocked` | reason `ContextNotFresh`; numerics `0` |
| budget invalid (`ContextWindowTokenLimit Ōēż 0`, `ReservedOutputTokenCount < 0`, `ReservedOutputTokenCount Ōēź ContextWindowTokenLimit`, or `ProviderCapabilityVersion Ōēż 0`) | block | `ContextBlocked` | reason `ModelBudgetUnavailable` |
| `Loaded` and `FullContextTokenCount Ōēż availableContextBudget` | full | `ContextReady` | `Mode=Full`, `UsedContextTokenCount=FullContextTokenCount`, budget + policy ref + capability version recorded (AC2) |
| `Loaded`, oversized, approved bounded behavior present + its limit fits | bounded | `ContextReady` | `Mode=Bounded`, `BoundedBehaviorReference` + bounds + `UsedContextTokenCount` + policy ref recorded (AC4) |
| `Loaded`, oversized, no approved bounded behavior | block | `ContextBlocked` | reason `ExceedsModelBudget`; `FullContextTokenCount` + budget recorded so audit shows the overflow (AC3 ŌĆö never silent) |

V1 (`"full-conversation-v1"`) ŌåÆ `ApprovedBoundedBehavior` is always `null`, so the bounded row is unreachable in production and the oversized case always blocks. [Source: PRD FR-9; ARCHITECTURE-SPINE.md#AD-10, #AD-11; ProviderCatalogEntryView.cs:35-42]
