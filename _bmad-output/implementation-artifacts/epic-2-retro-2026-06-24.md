## 2. Deep Story Analysis — Synthesized Patterns

### What went well

**A single, repeatable step-pattern carried the whole epic.** Every step was built the same way:

1. An **orchestrator** (Server, impure) does all I/O — reads, measurement, provider/Conversations calls — then dispatches **exactly one** command.
2. A **pure aggregate handler** (`AgentInteraction.Handle(...)`) emits events only (AD-3). The aggregate grew from 1 handler (2.1) to 5 (through 2.5) without ever reading another aggregate or doing I/O.
3. A **shared twin-policy** (`Evaluate` for the aggregate, `Decide` for the orchestrator's returned status) computes the decision **once**, so the aggregate and the orchestrator can never drift. `InternalsVisibleTo` exposes it to `*.Server` but deliberately **not** to `*.Server.Tests`, forcing tests to assert through the public surface.

This pattern, first crystallised in Story 2.2's gate, was reused almost verbatim for context (2.3), generation (2.4), and posting (2.5). It is the epic's biggest velocity multiplier and its most important reusable asset.

**Fail-closed-by-default is structural, not aspirational.** Every external dependency is bound to a `Deferred*` adapter that returns a safe not-available/unavailable result. The default DI graph **cannot** invoke a provider, read a Tenants/Agent read-model, evaluate content safety, or post to a Conversation — every unwired path resolves to `Denied` / `ContextBlocked` / `GenerationFailed` / `PostingFailed`. Live behaviour is opt-in behind config sections (`Parties`, `Conversations`); the only live adapters authored this epic are `PartiesAgentPartyDirectory` and the config-gated `ConversationClientContextReader` / `ConversationClientResponsePoster`.

**Sensitive content is confined by construction (AD-14).** Prompt and generated content live only on durable success events (`InteractionRequested`, `AgentOutputGenerated`) and aggregate state — never on commands, views, results, rejections, logs, badges, `aria-label`s, or `data-testid`s. This is enforced by assembly-wide `ContractsSecretNonDisclosureTests` plus per-story exact-name no-leak assertions.

**Quality discipline held the whole way.** 1281 green tests, zero build warnings, and adversarial Senior Developer Reviews on every story that found no correctness defects in production code (the one High was a fail-*open* risk caught in review and fixed with a regression test — see below).

### What was hard / where we struggled

- **Toolchain friction, repeatedly.** Parallel `dotnet build`/`restore` returned exit 1 with no diagnostics (fixed with `-m:1` serialized builds), and VSTest could not open its TCP listener in the sandbox (`SocketException (13)`), forcing the team to run built xUnit v3 executables directly. These were re-encountered and re-solved across stories.
- **Analyzer guard ergonomics.** The negative `state is not { IsRequested: true }` pattern was not recognised as a null-guard (CA1062), failing warnings-as-errors. The positive `state is { IsRequested: true } requested` pattern became the mandated idiom from 2.2 onward.
- **Re-reading context (2.3 → 2.4).** Story 2.3 deliberately discards raw conversation text after token measurement (only counts are kept), so Story 2.4 had to re-read content fresh to build model input — a non-obvious coupling surfaced during design.
- **Contract conformance edge cases.** "No custom attributes" assertions tripped on compiler-emitted `Nullable*Attribute`; "no content member" guards using substring matching tripped safe token-count fields. Both were re-scoped to exact-name / namespace-scoped matching.

### The one genuinely dangerous bug we caught

**Story 2.5, High severity — live membership verify failed *open* on a stale projection.** `ConversationClientResponsePoster.EnsureAiAgentParticipantAsync` inspected `ConversationDetailsV1.Participants` without first checking `Freshness.AllowsTrustBearingDecision()`. A visible-but-stale read could have carried a revoked-but-not-yet-projected membership as `Present` and allowed a post to proceed. Fixed by adding the trust-bearing guard (→ `ConversationUnavailable` when stale) and a dedicated regression test. This is the clearest proof that the adversarial review layer earns its keep — the fail-closed *intent* was right everywhere except the one place a live projection's freshness was assumed rather than checked.

### Recurring review feedback (a real pattern)

Across **all six** stories, the dominant Low-severity finding was the same: **the Dev Agent Record drifted from reality** — stale recorded test counts and File List omissions (a missing test file, a missing QA summary). It was auto-corrected every time, but it recurred every time. This is a process signal, not a code-quality signal, and it is the strongest candidate for a systemic improvement (see Action Items).

### Preparation gaps / risks

- **The live-binding backlog is real and concentrated.** Across Epics 1–2, *every* live external binding was deferred behind a `Deferred*` placeholder: read-models (Tenants projection, Agent/AgentInteraction readers), the `IAgentCommandDispatcher`, the provider-SDK adapter, the content-safety engine, the Conversations `AddParticipant` seam, and a runnable AppHost topology. The current plan lands most of this in **Epic 4** (4.1 API/client, 4.3 operational). Epic 3 can be fully built on the deferred pattern, but **nothing is end-to-end live yet**, and Epic 4 is accumulating disproportionate integration risk.
- **Conversations `AddParticipant` seam confirmed absent.** Story 2.5 verified the live Conversations module (commit `46df0cd`) exposes no public `AddParticipantAsync`; membership *establish* returns `SeamUnavailable` (fail-closed). The architecture already anticipated this in its Deferred table ("Story 2.0a prerequisite"). It does not block Epic 3, but it remains a prerequisite for any live automatic post.
- **OQ-10 (bounded-context behavior) still open.** 2.3 shipped the bounded-context *shape* but no approved bounded behavior; not on Epic 3's critical path, but unresolved.

## 8. Key Takeaways

1. **One disciplined step-pattern (orchestrator → single command → twin-policy) scaled across six stories** and is now the project's core reusable asset — write it down.
2. **Fail-closed by construction works** — the default DI graph is provably safe, and the only fail-*open* gap (2.5 stale projection) was caught by adversarial review.
3. **The recurring weak spot is record-keeping, not code** — Dev Agent Record drift recurred in every story and is the highest-value process fix.
4. **The bill for "deferred everything" comes due in Epic 4** — track the live-binding backlog now so integration risk doesn't surprise the team later.
